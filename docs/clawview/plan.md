### A. 计划摘要
本 v2 计划继续以“可观测性与风险发现”为唯一目标，按 **M1 指标口径冻结 -> M2 数据链路落地 -> M3 Display-Only 看板成型 -> M4 试运行与验收** 推进。  
相对 v1，v2 补齐 6 个阻塞项：  
1. 每个里程碑与阶段任务都明确了 `Owner + 截止时间`。  
2. 增加首发最小上线子集 **P0-Core（12 项）**，避免首发过重。  
3. 增加 `endpoint_group` 分组字典草案与治理规则。  
4. 增加重启分类优先级规则（Planned/Unexpected/Unknown）。  
5. 增加阈值冷启动校准机制（试运行后自动建议，人工确认生效）。  
6. 为验证门槛补齐量化阈值（覆盖率、延迟、准确性）。

### B. 范围与非范围（Scope / Non-Scope）
**Scope**
1. 首发上线按 **P0-Core（12 项）** 验收，保障“10 秒判断健康、30 秒定位风险”可用。
2. P0 全量字段（34 项）继续完成口径冻结与数据映射，但不作为首发阻塞项。
3. 双窗口统一口径：Rolling 24h + Asia/Tokyo 当日。
4. 首页调用次数固定为双口径展示：Rolling 24h（主）+ Tokyo 今日（辅）；周/月仅在 API 详情页展示。
5. 看板刷新延迟目标：P95 <= 15 分钟，P99 <= 20 分钟。
6. 风险发现聚焦三类：触发风暴、限速（429）、异常重启。
7. 看板信息架构固定为：顶部 KPI、中部趋势、侧边 Skill 模块、底部明细。
8. 首页只放轻量摘要，不放全量明细；明细统一下钻到子模块页（Skill/Cron/API/Error）。
9. Dashboard 首屏突出要素固定为 4 类：健康状态、重启风险、限速风险、错误风险。
10. 首页模块固定清单：健康总览（1 行）+ 趋势图（2 张）+ Skill/Cron/API 三个摘要模块。
11. Skill 模块首页仅展示摘要 + TopN（默认 5，可配置 6），全量 80~100+ Skill 通过详情页/抽屉下钻查看。
12. 原始快照 48h + 每日压缩快照长期保留。
13. 监控模式采用 Hook 触发式增量读取（cursor），v1 暂不加 cron 兜底。
14. 外发到用户后端是必需项，但必须执行白名单字段 + 本地脱敏 + TLS 传输。
15. v1 采用“够用优先”原则：优先保障趋势判断与异常发现，不追求 100% 精确。
16. 告警只做状态与解释，不触发控制动作。
17. 总数指标必须从个体组件聚合得出（先个体后总数）；TopN 与总数必须来自同一批聚合结果。

3. v1 首发不引入 P95/P99 之外的新高级性能指标作为阻塞项。
4. 不保留旧口径并行展示（避免双标导致误判）。
5. 不把 P0-Full（剩余 22 项）延期风险转嫁给首发验收。

### E. 指标字典冻结策略（P0/P1 与口径）
1. **唯一 SoT**：仅一份《ClawView v1 指标字典》生效；图表、告警、周报统一引用。
2. **发布分层**：  
   - 首发阻塞层：P0-Core 12 项。  
   - 首发非阻塞层：P0-Full 剩余 22 项，进入 v1.1 补齐。
3. **P0 冻结规则**：34 项都必须完成定义与口径，但首发验收以 P0-Core 为准。
4. **P1 管理规则**：只登记不阻塞上线，且不得改写 P0 公式。
5. **口径总则**：统一时区、统一窗口、统一计数边界、统一异常分类。
6. **版本策略**：字典版本化；任何变更必须附影响评估、生效时间、回滚条件。
7. **变更门禁**：先改字典，再改展示与告警；禁止先实现后补文档。
8. **API 限速统一口径**：HTTP 429 与各平台等价限流错误统一计入“限速次数（429）”，并用于“限速比例（429）”计算。

**endpoint_group 分组字典草案（v0.1）**
| group_key | 典型匹配范围（示意） | 备注 |
|---|---|---|
| auth | `/auth/*` `/login` `/token/*` | 鉴权相关 |
| account | `/users/*` `/profile/*` | 账号资料 |
| contacts | `/contacts/*` | 联系人 |
| conversations | `/conversations/*` `/threads/*` | 会话 |
| message_send | `/messages/send*` | 发送路径 |
| message_receive | `/messages/inbound*` `/messages/pull*` | 接收路径 |
| media | `/media/*` `/files/*` | 媒体与文件 |
| webhooks | `/webhooks/*` | 回调入口 |
| scheduler | `/jobs/*` `/scheduler/*` | 定时任务相关 |
| integrations | `/integrations/*` `/provider/*` | 第三方适配 |
| admin_config | `/admin/*` `/config/*` | 管理配置 |
| health_metrics | `/health` `/ready` `/metrics` | 健康与观测 |
| others | 未匹配项 | 收敛桶，不可拆成原始 URL |

**endpoint_group 治理规则**
1. 仅允许使用字典中的 group_key；展示层禁止原始 URL 维度。
2. 匹配优先级：`精确路由 > 最长前缀 > 关键词 > others`。
3. `others` 占比连续 2 天 > 5%，必须发起字典增补评审。
4. 字典新增/调整按周批处理，避免日常抖动导致维度爆炸。

**重启分类判定优先级（Planned/Unexpected/Unknown）**
| 判定顺位 | 分类 | 判定条件（满足即命中） |
|---|---|---|
| 1 | Unexpected | 存在高置信异常证据：崩溃、OOM、panic、非 0 退出、健康检查失败拉起、短时间连续重启 |
| 2 | Planned | 无 Unexpected 证据，且命中维护窗口/发布工单/人工批准重启，且停启序列符合预期 |
| 3 | Unknown | 证据不足或冲突无法消解 |

**冲突处理**
1. 同时命中 Planned 与 Unexpected 时，**Unexpected 优先**，并打上 `maintenance_overlap=true` 标签。
2. 证据源可信度优先级：运行时事件/编排器原因 > 发布系统/维护工单 > 人工备注。
3. Unknown 占比目标 <= 5%；连续 3 天 > 10% 视为 P1 问题并要求补证据链。

### G. 告警与阈值治理策略
1. v1 初始阈值以 PRD 已确认值为底线，不低于 PRD 下限。
2. 告警仅两级：黄灯（关注）/红灯（处置）。
3. 告警窗口与展示窗口一致，禁止跨窗口判断。
4. 数据质量前置门：数据缺失或延迟时，告警标注“可信度降低”。

**核心风险初始阈值（首发）**
| 风险 | 黄灯 | 红灯 | 窗口 | Owner |
|---|---|---|---|---|
| 触发风暴 | 同任务 5 分钟触发次数 >= 3 倍基线且 >= 20 | >= 5 倍基线且 >= 40 | 5m | SRE |
| 限速（429） | 限速比例（429） >= 3% | 限速比例（429） >= 8% | 15m | SRE |
| 异常重启 | Unexpected 重启 >= 1 次 | Unexpected 重启 >= 2 次或 10 分钟内连续 2 次 | 24h/10m | SRE |

**阈值冷启动校准机制（试运行后自动建议）**
1. 试运行采样窗口：至少 7 天连续样本。
2. 生效前提：单指标有效样本量 >= 1000 或覆盖完整 7 天窗口。
3. 自动建议规则：  
   - 比例类指标：`yellow_candidate = max(PRD_yellow, P95_stable)`，`red_candidate = max(PRD_red, P99_stable)`。  
   - 计数类指标：`yellow_candidate = max(PRD_yellow, median + 3*MAD)`，`red_candidate = max(PRD_red, median + 5*MAD)`。
4. 自动建议不自动生效，必须经过 `DA + SRE + PO` 评审确认。
5. 采纳条件：预估误报率下降 >= 20%，且召回率下降不超过 5%。
6. 调整频率：每周最多 1 次，保留前后值、证据、影响面、生效时间。

### H. 验证计划（如何验证计划执行质量）
1. **过程质量**：每个里程碑必须具备产出物、Owner、截止时间、评审记录。
2. **口径质量**：抽查关键指标，确保“字典定义=聚合结果=展示文案”一致。
3. **数据质量**：完整性、延迟、重复、异常值四类检查可复现。
4. **业务有效性**：按 AC 场景验收（10 秒健康判断、30 秒定位风险）。
5. **边界合规性**：逐条核对 Display-Only 清单，确认无控制能力泄漏。
6. **试运行质量**：跟踪告警准确性、看板可用性、阈值建议效果。
7. **复盘质量**：形成“问题-原因-改进”闭环并回灌下一轮批注。

**量化验收门槛（新增）**
| 维度 | 指标 | 验收阈值 | 阻塞级别 |
|---|---|---|---|
| 覆盖率 | P0-Core 指标覆盖率 | >= 98% | P0（阻塞） |

### J. 批注循环协议（v1->v2->v3）
1. **版本定义**：v1=初稿基线；v2=处理阻塞批注后版本；v3=上线前冻结候选版。
2. **批注输入格式**：`章节`、`问题级别(P0/P1/P2)`、`现象`、`建议`、`是否阻塞上线`。
3. **处理时限**：每轮批注 24 小时内回收并出“采纳/驳回/待定”决议表。
4. **升级规则**：存在任一 P0 阻塞不得升版；P1/P2 可进后续迭代。
5. **变更记录**：每次升版必须附变更摘要（新增/删除/修改/原因/影响里程碑）。
6. **轮次节奏**：建议最多 3 轮收敛；超过 3 轮仍有 P0 分歧则升级专项评审。
7. **冻结条件**：v3 要求“无 P0 阻塞、里程碑出口清晰、验证方案可执行”。
8. **冻结后规则**：新需求不直接改 v3，统一进入 v1.1 需求池，避免执行期 scope creep。
9. **责任归口**：P0 批注默认由对应章节 Owner 在 48 小时内给出闭环证据。
